# supabase/seeds — Production Data Policy

## ⛔ NO FAKE POLITICAL SEED DATA. EVER.

The political subsystem is a **live production system**. The following tables
must **never** be populated with fictional, sample, demo, or fixture data,
in any environment that touches real campaigns or real revenue:

- `political_routes`
- `political_organizations`
- `political_campaigns`
- `campaign_candidates`
- `political_outreach_leads`
- `political_plans`
- `political_scenarios`
- `political_route_selections`
- `political_reservations`
- `political_proposals`
- `political_orders`
- `political_contracts`
- `political_follow_ups`
- `political_approvals_log`

Fake rows in any of these tables corrupt sales pipelines, mislead operators,
contaminate reporting, and create real legal/compliance exposure
(FEC reporting, do-not-contact handling, USPS EDDM).

## Approved data sources

| Table                       | Approved source(s)                                                                                              |
| --------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `political_routes`          | USPS EDDM Retail facility tool CSV exports, imported via `/admin/political/routes/import`                       |
| `political_organizations`   | FEC committee bulk data (federal); Ohio Secretary of State PAC filings (state); manual admin entry              |
| `campaign_candidates`       | Manual entry by sales agents in `/admin/political/candidates`; or operator import from public race filings      |
| `political_campaigns`       | Created in-app when a sales agent engages a candidate                                                           |
| `political_outreach_leads`  | **Only** real submissions from the public `/political` portal, or admin-created leads tied to a real contact    |
| `political_plans`           | Created in-app for real, contracted campaign engagements                                                        |
| `political_scenarios`       | Generated by the quote engine for real plans                                                                    |
| `political_proposals`       | Generated in-app from a scenario the client has been shown                                                      |
| `political_orders`          | Created on contract signature + payment                                                                         |
| `political_contracts`       | Created on proposal acceptance                                                                                  |

## What seed files are allowed here?

Only **operational reference data** that is the same across every
environment and is not customer-, candidate-, or campaign-specific. Examples:

- Q&A knowledge base content (`qa_high_intent_pitch_scripts.sql`) — internal
  agent-facing answer scripts, not user data.
- Feature flag defaults.
- Reference enums/lookup tables that ship with the product.

If a file you want to add to this folder seeds rows into any table in the
list above, **do not add it**. Build an importer instead (see
`/admin/political/routes/import`, `/admin/political/organizations/import`).

## How to test the political subsystem locally

Use a separate, clearly-marked **non-production Supabase project** for
development. Either:

1. Run the importers against a **real CSV** you have legal rights to use
   (your own org's data, official FEC bulk download, etc.), OR
2. Hand-enter a small number of test rows directly via the Supabase Table
   Editor on your dev project, with a `notes` value of `'LOCAL DEV — DELETE BEFORE PROD'`,
   and never `pg_dump` that data into a shared environment.

Never copy local dev rows into the production database, and never check
local-dev SQL inserts into version control as seed files.
